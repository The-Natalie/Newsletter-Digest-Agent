# Feature: database.py + Alembic Baseline Migration

The following plan should be complete, but validate documentation and codebase patterns before starting.

Pay special attention to: SQLAlchemy Core (not ORM) table definitions, async engine setup, and Alembic's async env.py pattern.

## Feature Description

Create `database.py` — async SQLAlchemy engine, session factory, and Core table definition for `digest_runs` — then initialize Alembic and produce the baseline migration that creates the `digest_runs` table in SQLite.

## User Story

As the developer running the Newsletter Digest Agent,
I want a database layer and migration baseline committed to the repo,
So that any fresh checkout can run `alembic upgrade head` and have a ready database.

## Problem Statement

No database layer exists yet. The pipeline cannot persist digest runs, and there is no migration history for the `digest_runs` table defined in PRD §15.

## Scope

- In scope: `database.py` (engine + session factory + Core table), Alembic init + async `env.py`, baseline migration
- Out of scope: ORM models, additional tables, FastAPI lifespan wiring (done in `main.py` later), cross-run dedup tables

## Solution Statement

Use SQLAlchemy 2.0 `create_async_engine` with the `sqlite+aiosqlite` URL from `config.py`. Define `digest_runs` as a Core `Table` with a shared `MetaData`. Wire Alembic to use an async connection via `run_sync` so migrations run against the same async driver. Generate and apply the baseline migration.

### Patterns to Follow

**Engine creation (SQLAlchemy 2.0 async):**
```python
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession

engine = create_async_engine(settings.database_url, echo=False)
async_session = async_sessionmaker(engine, expire_on_commit=False)
```

**Core Table definition:**
```python
from sqlalchemy import MetaData, Table, Column, Text, Integer, DateTime
import sqlalchemy as sa

metadata = MetaData()

digest_runs = Table(
    "digest_runs",
    metadata,
    Column("id", Text, primary_key=True),
    ...
)
```

**Alembic env.py async pattern (key section to replace in generated env.py):**
```python
import asyncio
from sqlalchemy.ext.asyncio import create_async_engine

def run_migrations_online() -> None:
    connectable = create_async_engine(config.get_main_option("sqlalchemy.url"))

    async def run_async_migrations():
        async with connectable.connect() as connection:
            await connection.run_sync(do_run_migrations)
        await connectable.dispose()

    asyncio.run(run_async_migrations())
```

**alembic.ini sqlalchemy.url** — set to the async `sqlite+aiosqlite` URL. `env.py` reads `settings.database_url` (the same value) so the two are always in agreement:
```
sqlalchemy.url = sqlite+aiosqlite:///./data/digest.db
```

### TASK 5 — REPLACE `alembic/env.py` WITH ASYNC VERSION

- **IMPLEMENT**: Replace the generated `env.py` with an async-aware version that:
  1. Imports `metadata` from `database.py` and sets `target_metadata = metadata`

- **PATTERN**: Alembic async cookbook — `run_sync` bridge pattern (see documentation link above)
- **IMPORTS**: `asyncio`, `sqlalchemy.ext.asyncio.create_async_engine`, `from database import metadata`; `from config import settings`
- **GOTCHA**: `target_metadata = metadata` is required for `--autogenerate` to detect the `digest_runs` table

- **GOTCHA**: Do NOT call `settings = Settings()` inside env.py — import the already-instantiated `settings` from `config.py`
- **VALIDATE**: `alembic history` — this executes `env.py` in offline mode, confirming imports resolve and `target_metadata` is set. Expected output: `(no revisions)` (empty history is correct before generating the first migration).

**Full `alembic/env.py` content to write:**
```python
import asyncio
import os
import sys
from logging.config import fileConfig

from alembic import context
from sqlalchemy.ext.asyncio import create_async_engine

# Ensure project root is on sys.path so database.py and config.py are importable
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))

from config import settings
from database import metadata

# Alembic Config object (gives access to values within alembic.ini)
config = context.config

# Interpret the config file for Python logging
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

# The MetaData object from database.py — required for --autogenerate
target_metadata = metadata


def run_migrations_offline() -> None:
    """Run migrations in 'offline' mode (no live DB connection)."""
    url = settings.database_url
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


async def run_migrations_online() -> None:
    """Run migrations in 'online' mode using an async engine."""
    connectable = create_async_engine(settings.database_url, echo=False)
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()


if context.is_offline_mode():
    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())
```

## NOTES

**Why SQLAlchemy Core (not ORM)?**
CLAUDE.md explicitly specifies "SQLAlchemy Core (async)". The digest pipeline reads/writes entire JSON blobs — no object-relational mapping is needed. Core `Table` + raw SQL via `conn.execute(sa.text(...))` or `conn.execute(table.insert())` is simpler and sufficient.

**Why `expire_on_commit=False`?**
With `AsyncSession`, accessing ORM attributes after `commit()` would trigger lazy loads, which are not supported in async. Setting `expire_on_commit=False` avoids this. This is standard practice for all async SQLAlchemy sessions.

**Why does alembic.ini also have the URL if env.py reads from config.py?**
Both are set to the same `sqlite+aiosqlite:///./data/digest.db` value. `alembic.ini` is the entry point Alembic reads first; `env.py` then uses `settings.database_url` (the same value) to build the async engine. Keeping them in sync means `alembic` CLI tools that read `alembic.ini` directly (e.g. some IDE integrations) also see the correct URL. If `DATABASE_URL` changes in `.env`, update `alembic.ini` to match.

**Autogenerate reliability:**
`--autogenerate` works well for the initial migration when `target_metadata` is correctly set. For later migrations (Phase 2 new tables), always review the generated migration before applying.
